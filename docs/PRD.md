## 2. 产品概述

PC Repair Agent 是一款面向 PC 维修、装机、运维和普通用户故障修复场景的智能维修 Agent 工具。产品通过本地桌面 UI、Python 后台 Agent、可审查命令执行网关和可扩展 Skill 系统，帮助用户完成硬件识别、驱动下载、运行时环境补全、系统异常诊断与修复建议。

产品的核心差异点是：Agent 具备自动化能力，但不会无审查地执行高风险操作。所有涉及删除、移动、系统修改、环境变量修改、注册表修改、驱动安装、服务变更等高风险命令，必须先向用户解释用途和风险，并在用户确认后才允许执行。

## 3. 产品目标

### 3.1 核心目标

1. 降低 PC 维修和系统排障门槛。
2. 自动识别本机硬件、系统环境和缺失运行时。
3. 自动搜索并下载官方驱动和运行时组件。
4. 通过高风险命令审查机制提升用户信任。
5. 通过 Skill 系统沉淀可复用维修工作流。
6. 支持未来切换或并行使用 nanobot-sdk 与 codex-sdk。

### 3.2 非目标

1. MVP 阶段不追求全平台功能完全一致。
2. MVP 阶段不直接覆盖所有硬件厂商和所有驱动类型。
3. MVP 阶段不做无确认的全自动系统修复。
4. MVP 阶段不承诺替代专业维修工程师判断。

## 4. 目标用户

### 4.1 主要用户

| 用户类型 | 典型诉求 |
| --- | --- |
| PC 维修工程师 | 快速识别机器型号、驱动状态、系统异常，减少重复操作 |
| 装机店 / 电脑门店 | 为新机批量补齐驱动、运行库和常用组件 |
| IT 运维人员 | 快速排查办公电脑驱动、运行时、系统配置问题 |
| 普通用户 | 通过自然语言描述问题，获得可理解的修复方案 |

### 4.2 用户痛点

1. 不知道电脑具体型号、显卡型号、主板型号。
2. 搜索驱动容易进入第三方下载站，存在安全风险。
3. 不确定应该安装纯驱动还是厂商管理工具。
4. 缺少 VC++、.NET、DirectX、WebView2 等运行时导致软件无法启动。
5. 普通用户看不懂命令行操作，更不知道命令风险。
6. AI 工具如果直接执行系统命令，用户缺乏安全感。

## 5. 产品亮点

### 5.1 高风险命令审查

Agent 在执行命令前需要先进行风险分级。高风险命令必须进入用户确认流程，确认前不得执行。

高风险命令包括但不限于：

| 类型 | 示例 |
| --- | --- |
| 文件删除 | `del`, `Remove-Item`, `rd /s`, 清理系统目录 |
| 文件移动 / 覆盖 | 移动驱动文件、覆盖配置文件 |
| 环境变量修改 | 修改 `PATH`、系统级环境变量 |
| 注册表修改 | `reg add`, `reg delete`, `.reg` 导入 |
| 服务操作 | 启停、禁用、删除 Windows 服务 |
| 驱动安装 / 卸载 | 安装显卡、芯片组、网卡、蓝牙驱动 |
| 系统配置修改 | 防火墙、计划任务、启动项、组策略 |
| 权限提升 | 管理员权限、执行策略变更 |
| 下载后执行 | 从网络下载安装器后启动 |
| 磁盘相关 | 分区、格式化、启动配置、BitLocker 相关操作 |

审查弹窗必须展示：

1. 即将执行的命令或操作。
2. 操作目的。
3. 影响范围。
4. 风险点。
5. 是否可回滚。
6. 建议的备份或还原措施。
7. 用户确认按钮和取消按钮。

### 5.2 官方驱动下载

产品优先从设备厂商或整机厂商官网获取驱动，降低第三方下载站风险。

驱动下载前必须展示：

1. 设备名称。
2. 当前驱动版本。
3. 推荐驱动版本。
4. 来源厂商。
5. 官方下载 URL。
6. 发布日期。
7. 驱动类型：纯驱动 / 驱动加管理 UI。
8. 是否需要重启。

### 5.3 Skill 化维修工作流

核心能力以 Skill 形式呈现，便于用户理解和后续扩展。

MVP 阶段规划 Skill：

1. 全自动驱动下载安装工作流。
2. 笔记本驱动下载工作流。
3. 运行时环境补全工作流。
4. 智能疑难杂症诊断工作流。

## 6. 技术方案

### 6.1 推荐架构

```text
Tauri Desktop App
  - Web UI: React / Vue / Svelte + TypeScript
  - Rust Core: 权限网关、命令审批、进程管理、自动更新、sidecar 管理

Python Agent Backend
  - Agent Runtime Adapter
  - nanobot-sdk Adapter
  - codex-sdk Adapter
  - Skill Runner
  - 硬件识别、驱动搜索、运行时检测、诊断编排

Execution Gateway
  - 命令风险分级
  - 用户审批
  - 管理员权限执行
  - 审计日志
```

### 6.2 为什么选择 Tauri

1. 适合构建跨平台桌面可执行程序。
2. 包体相对 Electron 更轻。
3. Rust 后端适合作为本地安全边界。
4. 支持 sidecar，可以随应用携带 Python 后台程序。
5. 适合承载命令审批、权限隔离和进程管理。

### 6.3 为什么选择 Python 后台

1. Python 适合系统信息采集、脚本编排、驱动搜索、日志解析。
2. Windows WMI、注册表、设备信息、安装器自动化等生态更丰富。
3. 便于快速开发 Skill。
4. 可通过 PyInstaller / Nuitka 打包为 sidecar 可执行文件，用户无需单独安装 Python。

### 6.4 为什么保留 nanobot-sdk 与 codex-sdk

项目保留两个 SDK 的目的不是让业务逻辑直接依赖两个 SDK，而是通过 `Agent Runtime Adapter` 做隔离，便于后续切换、评估或按场景选择不同 Agent Runtime。

预期策略：

| SDK | 定位 |
| --- | --- |
| nanobot-sdk | 作为主要 Agent 编排候选，承载技能调用、任务规划和本地工作流 |
| codex-sdk | 作为可选 Agent Runtime，用于评估更强的代码/命令推理、任务拆解和本地自动化能力 |

业务层只依赖统一接口：

```text
AgentRuntime
  - plan(task)
  - run(skill, context)
  - propose_actions(context)
  - request_tool_call(tool, args)
  - summarize(result)
```

这样可以避免后续更换 SDK 时重写 UI、Skill 和命令审批系统。

### 6.5 通信方式

MVP 推荐：

1. Tauri 启动 Python sidecar。
2. Tauri 与 Python 通过 stdio、localhost HTTP 或 named pipe 通信。
3. Python 负责提出操作意图。
4. Tauri/Rust 负责执行前审查和用户确认。
5. 真正危险操作只由 Execution Gateway 执行。

## 7. 核心功能需求

### 7.1 首页 / 工作台

用户进入应用后看到当前电脑概览和推荐操作。

功能要求：

1. 展示系统版本、设备品牌、设备型号、CPU、GPU、主板、内存、磁盘。
2. 展示驱动健康状态。
3. 展示运行时环境状态。
4. 展示最近一次扫描时间。
5. 提供一键扫描入口。
6. 提供 Skill 快捷入口。

### 7.2 设备与系统扫描

系统扫描用于收集维修决策所需上下文。

扫描内容：

1. Windows 版本与构建号。
2. 设备厂商与型号。
3. 主板厂商与型号。
4. BIOS 版本。
5. CPU 型号。
6. GPU 型号与当前驱动版本。
7. 网卡、声卡、蓝牙设备。
8. 已安装运行时。
9. 关键系统日志摘要。
10. 是否为笔记本设备。

### 7.3 Skill：全自动驱动下载安装工作流

目标：自动识别硬件并从官网查找适配驱动。

流程：

1. 扫描硬件信息。
2. 判断需要更新的驱动类型。
3. 搜索官方驱动来源。
4. 解析驱动版本、发布日期和适用系统。
5. 向用户展示候选驱动。
6. 询问用户选择纯驱动或带 UI 的驱动套件。
7. 下载驱动安装包。
8. 校验文件签名或哈希。
9. 请求安装确认。
10. 执行安装。
11. 记录安装日志和结果。

MVP 优先支持：

1. NVIDIA 显卡驱动。
2. AMD 显卡驱动。
3. Intel 显卡 / 芯片组驱动。

### 7.4 Skill：笔记本驱动下载工作流

目标：针对笔记本优先从整机厂商官网下载驱动。

流程：

1. 判断设备是否为笔记本。
2. 识别品牌、型号、序列号或 SKU。
3. 打开对应品牌官网支持入口。
4. 匹配机型支持页面。
5. 获取推荐驱动列表。
6. 分类展示 BIOS、芯片组、显卡、声卡、网卡、蓝牙、电源管理、触控板等驱动。
7. 询问是否安装官方管理工具。
8. 下载并安装用户选择的驱动。

MVP 优先支持品牌：

1. Lenovo。
2. Dell。
3. HP。
4. ASUS。
5. Acer。
6. MSI。

### 7.5 Skill：运行时环境补全工作流

目标：检测常见运行时缺失并从官方来源安装。

检测项目：

1. Microsoft Visual C++ Redistributable。
2. .NET Desktop Runtime。
3. .NET Framework。
4. DirectX End-User Runtime。
5. Microsoft Edge WebView2 Runtime。
6. Java Runtime。

流程：

1. 扫描已安装运行时。
2. 识别缺失或版本过旧项目。
3. 展示推荐补全列表。
4. 展示官方来源和适用场景。
5. 下载官方安装包。
6. 请求用户确认安装。
7. 安装并记录结果。

### 7.6 Skill：智能疑难杂症诊断工作流

目标：用户用自然语言描述问题，Agent 自动收集信息并给出修复计划。

支持问题：

1. 游戏打不开或闪退。
2. 软件提示缺少 DLL。
3. 显卡驱动异常。
4. 蓝牙、声卡、网卡不可用。
5. 系统卡顿。
6. 蓝屏。
7. Windows 更新失败。
8. 外设识别异常。

流程：

1. 用户描述问题。
2. Agent 追问必要信息。
3. 自动收集相关系统上下文。
4. 输出可能原因排序。
5. 生成修复计划。
6. 对每一步涉及的命令或系统修改进行风险审查。
7. 用户确认后执行。
8. 展示结果和下一步建议。

### 7.7 命令审批中心

所有中高风险操作进入审批中心。

功能要求：

1. 展示待执行操作列表。
2. 每项操作展示风险等级。
3. 支持展开查看命令详情。
4. 支持逐条确认。
5. 支持本次任务全部取消。
6. 支持记住低风险偏好，但不得自动批准高风险命令。
7. 支持查看历史审批记录。

### 7.8 审计日志

产品必须记录所有重要操作。

日志内容：

1. 操作时间。
2. 操作来源 Skill。
3. Agent 建议内容。
4. 用户确认结果。
5. 执行命令。
6. 执行输出摘要。
7. 成功或失败状态。
8. 错误信息。
9. 回滚建议。

## 8. 安全与权限要求

### 8.1 权限原则

1. 默认最小权限运行。
2. 只有必要操作才请求管理员权限。
3. Python Agent 不直接执行高风险命令。
4. 高风险命令必须由 Execution Gateway 审查。
5. 下载文件必须标记来源并尽量校验签名。
6. 不从未知第三方站点下载驱动或运行时。

### 8.2 风险分级

| 等级 | 描述 | 是否需要确认 |
| --- | --- | --- |
| 低风险 | 只读信息采集、版本查询、无副作用检测 | 默认不需要 |
| 中风险 | 下载文件、打开安装器、启动诊断工具 | 需要提示或确认 |
| 高风险 | 修改系统、安装驱动、删除文件、修改环境变量 | 必须确认 |
| 禁止 | 格式化磁盘、破坏性删除、绕过安全策略 | 默认禁止，除非未来引入专家模式 |

### 8.3 审批展示规范

高风险操作必须使用用户可理解的语言描述。

示例：

```text
即将执行：安装 NVIDIA 显卡驱动 555.xx
用途：更新当前显卡驱动，解决游戏闪退和驱动版本过旧问题。
影响范围：显卡驱动、NVIDIA 控制面板、显示输出。
风险点：安装期间屏幕可能短暂黑屏；失败时可能需要重启或回滚驱动。
回滚方式：可在设备管理器中回退驱动，或使用系统还原点恢复。
```

## 12. 主要风险

### 12.1 技术风险

1. 不同品牌官网结构变化，驱动解析不稳定。
2. Python sidecar 打包体积较大。
3. 杀毒软件可能拦截驱动下载或安装行为。
4. Windows 权限模型复杂，管理员权限交互需要设计清楚。
5. SDK 之间能力差异可能导致适配层抽象不足。

### 12.2 产品风险

1. 用户过度信任自动修复，忽视风险提示。
2. 驱动安装失败可能导致显示、网络等关键功能异常。
3. 普通用户难以理解驱动版本和运行时差异。
4. 专业用户可能需要更细粒度控制。

### 12.3 合规与安全风险

1. 下载源必须尽量使用官方链接。
2. 不应绕过厂商许可或安装限制。
3. 不应默认收集或上传隐私数据。
4. 日志中需要避免记录敏感信息。
